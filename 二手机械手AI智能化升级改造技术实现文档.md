# 二手机械手 AI 智能化升级改造技术实现文档

## 1. 文档目的

本文档面向项目研发人员、合作开发工程师及技术实施人员，用于明确本项目的技术实现边界、系统架构、模块分工、接口方式、阶段开发重点与联调流程，作为项目研发落地的基础技术说明。

## 2. 项目目标

本项目目标是基于现有二手机械手本体、伺服系统、正运动 ZMC406 运动控制器和 Motion SDK 通讯能力，构建一套可扩展的智能机械手系统，实现以下能力：

- 支持机械手底层精准运动控制
- 支持上位机统一任务调度和状态管理
- 支持语音输入和语义转控制指令
- 支持视觉识别、目标定位和安全检测
- 支持大模型任务理解与动作规划
- 支持通过标准接口完成 AI 模块与运动控制模块解耦

项目总体遵循“底层控制稳定优先、上层智能逐步增强、先闭环后优化”的技术原则。

## 3. 总体技术路线

系统采用分层架构设计：

1. 设备执行层
2. 运动控制层
3. 通讯接口层
4. 上位机调度层
5. AI 感知与决策层

控制主链路如下：

1. 摄像头、麦克风采集现场数据
2. 语音模块、视觉模块、大模型模块输出结构化结果
3. 上位机解析任务并转换为标准动作调用或轨迹参数
4. 上位机通过 Motion SDK 与 ZMC406 通讯
5. ZMC406 根据预设动作库或轨迹参数执行机械手动作
6. 运行状态、报警信息、任务结果回传上位机

技术上不建议让 AI 模块直接操作底层伺服控制，而应采用“AI 输出任务，上位机负责解析，控制器负责执行”的分层方式，以保障实时性、稳定性和安全性。

## 5. 模块职责划分

### 5.1 底层控制模块

负责内容：

- 机械手回零、点位运动、多轴联动
- 抓取、放置、搬运、码垛等固定动作
- 插补、轨迹执行、速度控制、加减速控制
- IO 输入输出控制
- 限位、急停、互锁等安全逻辑

建议输出接口：

- `home()`
- `stop()`
- `reset()`
- `pick()`
- `place()`
- `move_to_pose(x, y, z, rx, ry, rz)`
- `run_action(action_name, params)`
- `get_status()`

### 5.2 Motion SDK 封装模块

负责内容：

- 控制器连接和断开
- 命令发送与状态读取
- 错误码处理
- SDK 二次封装
- 为上位机提供统一控制接口

建议封装方向：

- 底层保留官方 API 调用
- 上层暴露项目统一接口
- 屏蔽 AI 工程师对控制器底层细节的直接依赖

### 5.3 上位机任务调度模块

负责内容：

- 接收语音、视觉、大模型结果
- 将结构化任务转换为控制动作
- 任务排队、执行、暂停、恢复、终止
- 调用 Motion SDK
- 记录执行结果和异常信息

建议能力：

- 支持手动模式与自动模式切换
- 支持任务优先级
- 支持单步调试和全自动运行

### 5.4 视觉模块

负责内容：

- 摄像头采集
- 图像预处理
- 目标识别
- 目标定位
- 安全区域检测
- 可选的三维空间感知

第一阶段建议：

- 优先做二维定位
- 优先处理规则物料
- 优先实现目标存在判断和简单坐标输出

### 5.5 语音模块

负责内容：

- 麦克风音频采集
- 降噪
- 语音识别
- 简单命令词识别
- 语义结构化

第一阶段建议：

- 使用有限命令集
- 不追求自由对话
- 优先实现“开始、停止、抓取、复位”等固定指令

### 5.6 大模型模块

负责内容：

- 自然语言任务理解
- 任务拆解
- 目标选择
- 动作规划
- 输出结构化任务数据

第一阶段建议：

- 大模型只负责任务理解
- 不直接生成底层控制代码
- 必须通过固定 JSON 协议向上位机输出任务

### 5.7 UI 界面模块

负责内容：

- 设备连接状态显示
- 当前任务显示
- 实时视频显示
- 动作按钮控制
- 报警信息显示
- 运行日志查看

建议首版页面：

- 首页状态总览
- 手动控制页
- 自动任务页
- 报警日志页
- 参数配置页

## 6. 推荐技术栈

### 6.1 当前推荐方案

- 上位机主程序：Python
- 控制器接口封装：Python 优先，必要时补 C++
- UI：PyQt6
- 视觉：OpenCV
- 语音：Python 语音识别接口
- 大模型对接：Python
- 高性能模块：后续按需转 C++

### 6.2 推荐原因

- Python 更适合快速接入 AI、语音、视觉能力
- PyQt6 适合快速搭建工业界面
- 控制器高实时部分继续由 ZMC406 负责，降低上位机实时性压力
- 后期性能瓶颈模块可逐步替换为 C++ 实现

## 7. 接口设计建议

## 7.1 模块间数据流

建议数据流：

`语音/视觉/大模型 -> 结构化任务 -> 上位机调度 -> Motion SDK 封装 -> ZMC406 -> 机械手执行`

### 7.2 统一任务协议

建议所有 AI 模块最终输出统一 JSON 数据结构。

示例：

```json
{
  "task_id": "task_0001",
  "task_type": "pick_and_place",
  "target_id": "part_01",
  "pick_point": [120.5, 230.0, 50.0],
  "place_point": [410.0, 150.0, 60.0],
  "pose": [0.0, 90.0, 0.0],
  "speed": 80,
  "priority": "normal",
  "safety_check": true,
  "source": "voice+vision+llm"
}
```

### 7.3 上位机内部动作接口

建议上位机统一提供如下服务接口：

```python
connect_controller()
disconnect_controller()
run_action(action_name, params)
move_to_pose(x, y, z, rx, ry, rz)
pause_task(task_id)
resume_task(task_id)
stop_task(task_id)
emergency_stop()
get_controller_status()
get_task_status(task_id)
```

### 7.4 视觉输出协议建议

```json
{
  "camera_id": "cam_01",
  "target_found": true,
  "target_id": "part_01",
  "position": [120.5, 230.0],
  "angle": 35.2,
  "confidence": 0.94,
  "safe_region_ok": true,
  "timestamp": "2026-03-25T20:00:00+08:00"
}
```

### 7.5 语音输出协议建议

```json
{
  "text": "抓取左边托盘里的工件放到右边工位",
  "intent": "pick_and_place",
  "target_area": "left_tray",
  "destination_area": "right_station",
  "confidence": 0.91,
  "timestamp": "2026-03-25T20:00:00+08:00"
}
```

## 8. 开发阶段规划

### 8.1 第一阶段目标

目标是做出可运行的最小可行系统，不追求复杂 AI，自上而下先闭环。

第一阶段必须实现：

- 机械手基础动作稳定执行
- Motion SDK 能稳定收发命令
- 上位机能调用固定动作
- UI 可查看状态并手动触发
- 视觉模块能识别目标是否存在
- 语音模块能识别有限命令词
- 全链路完成一次完整任务闭环

### 8.2 第一阶段开发任务拆分

#### A. 底层控制组

- 完成 ZMC406 连接与调试
- 完成回零、抓取、放置、搬运动作
- 完成急停、复位、安全互锁
- 输出动作调用规范

#### B. 上位机开发组

- 完成项目主程序框架
- 完成 SDK 封装
- 完成 UI 初版
- 完成任务调度和状态显示

#### C. 视觉开发组

- 完成摄像头接入
- 完成图像采集
- 完成简单目标检测和定位
- 输出结构化识别结果

#### D. 语音/AI 开发组

- 完成语音采集和识别
- 完成简单命令解析
- 将语音结果转成统一协议

### 8.3 第二阶段开发重点

- 增加目标精定位
- 增加动态轨迹参数输入
- 支持可变抓取点
- 增加避障和安全联动
- 优化任务调度与异常恢复

### 8.4 第三阶段开发重点

- 增加连续轨迹和高精度插补控制
- 增加视觉实时修正
- 增加大模型预测与多模态融合
- 适配焊接、精细分拣等高精度场景

## 9. 第一版最小可行系统建议

第一版建议控制范围如下：

- 1 台机械手
- 1 套 ZMC406 控制系统
- 1 台工控机
- 1 个摄像头
- 1 个麦克风
- 1 套 PyQt6 界面
- 3 到 5 个底层固定动作
- 2 到 3 条语音命令
- 1 个视觉识别任务
- 1 个可演示应用场景

推荐演示场景：

- 工件抓取搬运
- 机床上下料
- 规则物料码垛

原因：

- 场景标准化程度高
- 识别难度低
- 动作逻辑清晰
- 容易形成演示效果

## 10. 软件工程建议

### 10.1 代码结构建议

建议按如下方式组织项目：

```text
project/
  app/
    main.py
    ui/
    controllers/
    scheduler/
    services/
    vision/
    voice/
    llm/
    sdk/
    models/
    logs/
    configs/
```

### 10.2 设计原则

- 控制逻辑与 AI 逻辑解耦
- 所有模块通过统一协议交互
- 动作调用尽量幂等
- 异常必须可记录、可追踪、可恢复
- 手动模式优先于自动模式
- 急停和安全逻辑必须独立于 AI 输出

### 10.3 日志建议

至少记录以下内容：

- 控制器连接日志
- 指令发送日志
- 动作执行日志
- 视觉识别结果日志
- 语音识别日志
- 报警日志
- 急停和异常中断日志

## 11. 联调流程建议

推荐按以下顺序联调：

1. 机械手底层动作单独调通
2. Motion SDK 与控制器单独调通
3. 上位机手动按钮调用动作调通
4. UI 状态显示和日志调通
5. 视觉结果接入并触发固定动作
6. 语音结果接入并触发固定动作
7. 语音 + 视觉 + 控制完整闭环联调
8. 大模型任务解析接入
9. 多模块异常测试和安全测试

不建议一开始就把所有模块一起开发后再总联调，那样定位问题会非常困难。

## 12. 风险点与技术注意事项

### 12.1 实时性问题

AI 推理可能存在延迟，控制实时性不能依赖云端或高耗时推理。实时控制动作必须由控制器和预封装动作库完成。

### 12.2 安全问题

AI 输出结果可能错误，所有动作执行前应增加边界检查、姿态校验、安全区判断和执行许可判断。

### 12.3 坐标转换问题

视觉坐标系与机械手坐标系之间需要完成标定和转换，否则无法稳定抓取。该部分是视觉联动成败关键。

### 12.4 工业现场干扰

噪声、光照变化、遮挡、设备振动和网络抖动都可能影响识别和控制效果，第一阶段应优先在受控环境下验证。

## 13. 当前最急需补齐的技术工作

按现阶段情况，建议优先补齐以下工作：

- Motion SDK 项目级封装方案
- Python/PyQt6 上位机主框架
- 统一任务 JSON 协议
- 视觉坐标输出与映射方案
- 语音命令解析流程
- AI 工程师与控制工程师的接口边界定义

## 14. 结论

本项目的核心不是单独开发语音、视觉或机械手控制，而是建立一套稳定的“AI 感知决策 + 上位机调度 + Motion SDK 通讯 + ZMC406 精准执行”的分层系统。

从研发实施角度看，最正确的推进方式是先建立最小闭环系统，先做固定动作调用，再逐步增加视觉定位、动态轨迹和大模型任务规划能力。只有这样，项目才能在研发风险、交付节奏和商业落地之间取得平衡。
