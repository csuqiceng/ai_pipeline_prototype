## 当前状态判断

项目已经从“纯软件原型”推进到“可进入真机前联调准备”的阶段。

现在的问题不再是“有没有 SDK 接口壳子”，而是：

1. 没有真机，无法完成最终控制器联调
2. 安全、异常、测试体系还不够完整
3. 无界面 smoke test 仍然和 GUI 导入耦合
4. 真机动作接口路线还没最终定成“点位式”还是“固定动作式”

## 现在最该做

在没有真机的前提下，优先做下面四类工作：

1. 补齐异常处理和安全逻辑
2. 补自动化测试
3. 解耦 GUI 和无界面 smoke test
4. 标准化视觉/语音输入接口，为后续真实接入做准备

## 本周建议顺序

### 1. 安全和异常

目标：

- 急停后禁止继续下发动作
- 报警激活时拦截任务执行
- 控制器未连接时拦截 SDK 执行器动作
- 执行失败时自动写入报警与错误原因
- 界面和应用服务层能看到明确状态变化

对应任务：

- `C-07` 急停与复位逻辑
- `C-08` 状态读取接口
- `P-07` 异常处理机制
- 第一阶段报警与安全模块

建议先改：

- `ai_pipeline_prototype/dispatcher.py`
- `ai_pipeline_prototype/controller_service.py`
- `ai_pipeline_prototype/executor.py`
- `ai_pipeline_prototype/app_service.py`

### 2. 自动化测试

目标：

- 给当前原型补基础测试，防止后续改真机逻辑时把主链路改坏
- 覆盖成功流、失败流、越界、安全拦截和状态刷新

对应任务：

- `T-01` 单模块测试
- `T-03` 异常测试

建议先补：

- `planner` 测试
- `dispatcher` 测试
- `controller_service` 测试
- `sdk_adapter` mock 测试

### 3. 无界面联调入口解耦

目标：

- 让无界面环境也能做 smoke test
- 不再依赖 `tkinter` 导入成功才能运行冒烟检查

对应任务：

- 第一阶段工具链稳定性补齐

建议做法：

- 把 `gui.py` 里的 smoke test 拆到独立模块
- 或延后 `tkinter` 导入时机

### 4. 动作接口路线预研

目标：

- 在真机到位前，把执行器接口收敛好
- 提前决定后续更适合“点位式”还是“固定动作式”

判断标准：

- 如果后续直接下发位置和 IO，更适合点位式
- 如果控制器现场已有 BAS 程序、动作号、固定工艺流程，更适合固定动作式

当前倾向：

- 先保持现在的点位式接口
- 等真机联调后再最终拍板

### 5. 视觉和语音接口标准化

目标：

- 在没接真实设备前，先把输入协议和错误处理定下来
- 避免后续摄像头、麦克风接入时推翻上层接口

对应任务：

- `V-06` 输出协议对接
- `S-04` 语义结构化
- `S-05` 协议输出

## 下一步最关键的一件事

优先补齐“安全/异常逻辑 + 自动化测试”。

原因：

- 这部分完全不依赖真机，可以立即推进
- 真机到位前，最容易提升项目质量的就是这两块
- 如果没有这两块，后续接真实控制器时风险会很高
- 这一步做完后，项目会从“能演示”变成“更可维护、可联调、可扩展”
