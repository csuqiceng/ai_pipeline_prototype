# 给 AI 工程师的对接接口说明书

## 1. 文档目的

本文档用于帮助 AI 工程师快速理解本项目中的对接边界、数据接口、输出要求和联调方式，确保 AI 模块能够顺利接入我方上位机控制系统，而无需介入底层运动控制实现。

## 2. 项目协作边界

### 2.1 我方负责内容

- 机械手本体及伺服系统调试
- ZMC406 运动控制程序开发
- ZBasic 动作库封装
- Motion SDK 通讯封装
- Python/C++ 上位机控制主框架
- 动作执行、安全互锁、急停和报警逻辑

### 2.2 AI 工程师负责内容

- 语音识别与语义理解
- 视觉识别、检测、定位和安全检测
- 大模型任务理解、任务拆解和动作参数生成
- 将 AI 结果按约定格式传递给我方上位机

### 2.3 协作原则

AI 模块不直接操作控制器，不直接下发底层伺服指令。  
AI 模块只负责输出结构化结果，由我方上位机进行协议解析、任务调度和 Motion SDK 调用，再由 ZMC406 执行机械手动作。

## 3. 总体对接流程

标准流程如下：

1. 语音模块输出用户指令或结构化意图
2. 视觉模块输出目标识别结果、位置和安全状态
3. 大模型或任务规划模块整合信息，输出标准任务 JSON
4. 上位机接收 JSON，做字段校验和动作映射
5. 上位机通过 Motion SDK 调用控制器接口
6. 控制器执行动作并返回状态

## 4. AI 工程师需要理解的核心原则

### 原则 1

底层运动控制由我方负责，AI 不需要处理伺服、插补、控制器寄存器、IO 电平等细节。

### 原则 2

AI 输出必须是结构化、可校验、可追踪的数据，不能仅返回自然语言文本。

### 原则 3

AI 输出的是“任务和参数”，不是“底层执行代码”。

### 原则 4

所有输出必须经过字段完整性检查和边界值检查后，才能进入执行链路。

## 5. AI 模块输入信息

AI 模块通常会接收以下信息：

- 语音识别文本
- 视觉识别结果
- 目标位置信息
- 当前设备状态
- 当前任务状态
- 历史任务上下文

我方后续可根据项目需要提供以下信息给 AI：

- 当前机械手状态
- 是否空闲
- 当前工作模式
- 当前报警状态
- 当前允许动作列表

## 6. AI 模块输出要求

AI 工程师最终需要输出统一 JSON 数据，供我方上位机解析。

### 6.1 标准任务协议

```json
{
  "task_id": "task_0001",
  "task_type": "pick_and_place",
  "target_id": "part_01",
  "target_area": "left_tray",
  "destination_area": "right_station",
  "pick_point": [120.5, 230.0, 50.0],
  "place_point": [410.0, 150.0, 60.0],
  "pose": [0.0, 90.0, 0.0],
  "speed": 80,
  "priority": "normal",
  "safety_check": true,
  "source": "voice+vision+llm",
  "confidence": 0.92,
  "timestamp": "2026-03-25T20:00:00+08:00"
}
```

### 6.2 字段说明

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| `task_id` | string | 是 | 任务唯一编号 |
| `task_type` | string | 是 | 任务类型，如 `pick_and_place` |
| `target_id` | string | 否 | 目标对象编号 |
| `target_area` | string | 否 | 目标所在区域 |
| `destination_area` | string | 否 | 目标放置区域 |
| `pick_point` | array | 否 | 抓取点坐标 |
| `place_point` | array | 否 | 放置点坐标 |
| `pose` | array | 否 | 抓取姿态或末端姿态 |
| `speed` | number | 否 | 运动速度建议值 |
| `priority` | string | 否 | 任务优先级 |
| `safety_check` | boolean | 是 | 是否要求执行前安全检查 |
| `source` | string | 否 | 数据来源说明 |
| `confidence` | number | 否 | AI 判断置信度 |
| `timestamp` | string | 是 | 输出时间 |

## 7. 可接受的任务类型建议

第一阶段建议 AI 输出以下任务类型：

- `start_task`
- `stop_task`
- `reset_robot`
- `go_home`
- `pick`
- `place`
- `pick_and_place`
- `inspect_target`

第二阶段可扩展：

- `dynamic_pick`
- `dynamic_place`
- `sort_by_rule`
- `vision_guided_pick`

第三阶段可扩展：

- `tracking_pick`
- `fine_assembly`
- `continuous_path_task`

## 8. 语音模块接口规范

### 8.1 输出示例

```json
{
  "text": "抓取左边托盘里的工件放到右边工位",
  "intent": "pick_and_place",
  "target_area": "left_tray",
  "destination_area": "right_station",
  "confidence": 0.91,
  "timestamp": "2026-03-25T20:00:00+08:00"
}
```

### 8.2 最低要求

- 能识别固定命令词
- 能提取意图
- 能提取目标区域或对象信息
- 能输出置信度
- 能带时间戳

### 8.3 第一阶段建议命令集

- 开始作业
- 停止作业
- 机械手回零
- 抓取目标
- 放置目标
- 系统复位

## 9. 视觉模块接口规范

### 9.1 输出示例

```json
{
  "camera_id": "cam_01",
  "target_found": true,
  "target_id": "part_01",
  "target_type": "metal_part",
  "position": [120.5, 230.0],
  "angle": 35.2,
  "confidence": 0.94,
  "safe_region_ok": true,
  "timestamp": "2026-03-25T20:00:00+08:00"
}
```

### 9.2 最低要求

- 能判断目标是否存在
- 能输出目标位置
- 能输出识别置信度
- 能输出安全区状态

### 9.3 第二阶段可增加

- 三维坐标
- 姿态角
- 多目标排序
- 障碍物信息

## 10. 大模型 / 任务规划模块接口规范

### 10.1 目标

将语音意图、视觉结果、当前状态整合为统一任务对象。

### 10.2 输出要求

- 不输出底层控制代码
- 不输出控制器内部指令
- 不直接调用 Motion SDK
- 只输出结构化任务和参数

### 10.3 推荐流程

1. 读取语音意图
2. 读取视觉结果
3. 判断任务是否有效
4. 判断目标是否存在
5. 判断是否具备执行条件
6. 生成统一任务 JSON

## 11. 我方上位机预留接口

我方上位机后续将提供以下能力供联调：

- 接收 JSON 任务输入
- 校验字段完整性
- 判断当前设备是否允许执行
- 将任务映射为固定动作或参数化动作
- 返回任务接收状态
- 返回执行中状态
- 返回执行成功或失败结果

示例响应：

```json
{
  "task_id": "task_0001",
  "accepted": true,
  "controller_ready": true,
  "message": "task accepted",
  "timestamp": "2026-03-25T20:00:05+08:00"
}
```

执行结果响应示例：

```json
{
  "task_id": "task_0001",
  "status": "completed",
  "result": "success",
  "error_code": 0,
  "message": "pick and place finished",
  "timestamp": "2026-03-25T20:00:15+08:00"
}
```

## 12. 联调建议

建议按以下顺序对接：

1. 先对接固定 JSON 文件输入
2. 再对接本地接口调用
3. 再对接实时语音模块
4. 再对接实时视觉模块
5. 最后接入大模型综合任务规划

这样做的原因是可以先确保协议正确，再逐步增加系统复杂度。

## 13. AI 工程师常见误区

### 误区 1

直接让大模型输出底层控制指令。  
不建议。底层控制必须由我方统一管理。

### 误区 2

输出自然语言说明，不输出结构化数据。  
不建议。上位机需要稳定可解析的数据格式。

### 误区 3

未做置信度与安全判断就直接下发任务。  
不建议。工业场景必须保留校验与拒绝机制。

### 误区 4

将 AI 模块与控制器强绑定。  
不建议。后期要支持多场景和多设备，必须保持模块解耦。

## 14. 第一阶段最小对接目标

AI 工程师第一阶段只需完成以下内容：

- 语音识别固定命令词
- 输出结构化语义结果
- 视觉识别目标是否存在
- 输出目标位置
- 合成标准任务 JSON
- 交由我方上位机执行

第一阶段不要求：

- 完整自由对话
- 高精度实时轨迹控制
- 复杂多目标优化
- 直接控制机械手底层运动

## 15. 结论

AI 工程师在本项目中的主要工作是“感知、理解、生成任务数据”，而不是“直接控制机械手”。  
只要 AI 模块能够稳定输出符合协议的结构化结果，我方即可通过上位机与 Motion SDK 将其转换为机械手的实际动作执行。
