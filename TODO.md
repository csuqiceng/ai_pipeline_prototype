## 当前状态判断

项目已经从“纯软件原型”推进到“可演示、可联调、已接入真实语音能力的原型阶段”。

现在的核心事实是：

1. 软件主链路已经闭环，CLI 和 GUI 都能走完整流程
2. 讯飞 IAT 已接入，文件音频和麦克风输入都已经能进入当前项目
3. 真机控制链路已有 SDK 适配和联调入口，但还没有完成稳定的实机动作闭环
4. 当前最大的风险已经从“能不能接上能力”转为“安全、异常、测试、真机准备是否到位”

## 现在最该做

在没有真机长期可用的前提下，优先做下面四类工作：

1. 补齐安全和异常拦截
2. 补自动化测试
3. 收稳 GUI / 语音链路
4. 为真机联调整理参数和接口边界

## 本周建议顺序

### 1. 安全和异常

目标：

- 急停后禁止继续下发动作
- 报警激活时拦截任务执行
- 控制器未连接时拦截 SDK 执行器动作
- 执行失败时自动写入报警与错误原因
- 界面和应用服务层能看到明确状态变化

对应任务：

- `C-07` 急停与复位逻辑
- `C-08` 状态读取接口
- `P-07` 异常处理机制
- 第一阶段报警与安全模块

建议先改：

- `ai_pipeline_prototype/dispatcher.py`
- `ai_pipeline_prototype/controller_service.py`
- `ai_pipeline_prototype/executor.py`
- `ai_pipeline_prototype/app_service.py`

### 2. 自动化测试

目标：

- 给当前原型补基础测试，防止后续改真机逻辑时把主链路改坏
- 覆盖成功流、失败流、越界、安全拦截和状态刷新
- 补语音规则测试，避免命令词扩展时回归退化

对应任务：

- `T-01` 单模块测试
- `T-03` 异常测试

建议先补：

- `planner` 测试
- `dispatcher` 测试
- `controller_service` 测试
- `sdk_adapter` mock 测试
- `inputs.py` 语音规则测试

### 3. GUI / 语音链路稳定性

目标：

- 让 GUI 麦克风识别更稳定、更容易排查问题
- 降低“识别为空”“设备选错”“录到静音”时的定位成本

建议做法：

- 增加“保存本次麦克风录音”的调试开关
- 识别失败时给出更明确的 GUI 提示
- 在 GUI 中补充推荐命令词示例
- 继续扩展口语规则，但保持规则可解释、可测试

### 4. 真机联调前准备

目标：

- 在真机到位前，把执行器接口和现场参数准备好
- 明确后续更适合“点位式”还是“固定动作式”

需要提前确认：

- 控制器连接方式
- 轴号
- 回零模式
- 停止模式
- 夹具输出口
- 现场是否已有 BAS / ZBasic 固定动作流程

当前倾向：

- 保持现在的点位式接口不动
- 等真机联调后再决定是否切到固定动作式
